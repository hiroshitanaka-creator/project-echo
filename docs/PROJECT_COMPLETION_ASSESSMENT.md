## 3) 残タスクのPhase計画（現実的な分割）

### Phase A（1〜2週間）: 進捗と品質ゲートの同期
**目的**: 「何が完了か」を誰でも同じ判定にする。

- PROGRESS / README / CHANGELOG の整合化（P1実績反映、未完項目を明示）。
- P1のDoD（Definition of Done）を固定化（必須テスト・必須証跡・責任境界）。
- CIの必須ゲートを文書と実運用で一致させる（smoke/invariants/prop-core の失敗時運用を固定）。

**完了条件**
- 単一の進捗ソースで、P1達成率が説明可能。
- 主要ドキュメント間で矛盾がない。

### Phase B（2〜4週間）: P1残件の完了宣言に向けた詰め
**目的**: 実装済み機能を「運用可能品質」に引き上げる。

- 音声系 property-based test の不足領域を洗い出して追加。
- Demo C の再現手順をCIまたは定期ジョブで検証（手順の“実行可能性”担保）。
- Echo Mark鍵運用（rotate/revoke）を運用Runbookとして演習ベースで確認。

**完了条件**
- P1完了を客観的に示すテスト/証跡セットが揃う。
- 新規参加者が手順通りに再現可能。

### Phase C（1〜2ヶ月）: P2（統合運用テンプレート & 監査自動化）
**目的**: 公開監査に耐える日次運用へ移行。

- 監査レポート生成（署名付き）を定常ジョブ化。
- 週次benchmark判定を“是正まで含む運用”へ昇格。
- インシデント時の責任境界テンプレート（技術責任/運用責任）を標準化。

**完了条件**
- 「検証・監査・是正」が1サイクルとして回る。
- KPI逸脱時の対応時間/復旧条件が定義される。

### Phase D（1〜2ヶ月）: v1.0リリース準備
**目的**: “実装がある”状態から“継続提供できる”状態へ。

- v1.0リリース基準（互換性、セキュリティ、監査公開範囲）を固定。
- 変更管理（breaking change方針、移行ガイド、鍵運用規約）を整理。
- 公開監査レポートの定期発行体制を確立。

**完了条件**
- v1.0判定会で、実装・運用・監査の3軸を同時に満たせる。
