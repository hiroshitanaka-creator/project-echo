# Web Security Audit Report (Zero-Tolerance)

Date: 2026-04-23 (UTC)
Repository: `hiroshitanaka-creator/project-echo`
Scope: Full repository inventory of actual Web/HTTP attack surface and adjacent input boundaries.

## 1) Attack Surface Inventory

### Inbound HTTP / browser-facing entrypoints
1. `server.js` (Node native HTTP server)
   - `GET /` returns static JSON capability description.
   - `GET /health` returns static health JSON.
   - `GET /api/voice/schema` invokes a fixed Python one-liner (`spawnSync`) and returns JSON.
   - `POST /api/voice/run` parses JSON request body and always returns structured `501 not_implemented`.

2. `tools/demo_web.py` (Gradio demo UI)
   - Browser-facing local demo app (`app.launch(...)`) with optional remote bind by env gate `ECHO_DEMO_ALLOW_REMOTE=1`.
   - User-controlled fields: `intent` and `transcript` textbox values.
   - UI output paths: text/code panels and one HTML panel (`gr.HTML`) generated by `pig_panel()` from fixed message templates.

### Outbound HTTP
1. `src/po_echo/webhook_dispatch.py`
   - Slack webhook delivery and PagerDuty Events API delivery via `urllib.request.urlopen`.
   - No inbound request handling.

2. `scripts/send_webhook_alert.py`
   - CLI wrapper for outbound webhook dispatch only.

### Security-relevant config / docs / examples
1. `.env.example`
   - Placeholder-only secrets for demo startup.
2. `README.md` web sections
   - Demo launch instructions and Node gateway behavior.
3. `tools/demo_shopping.py`
   - Not a web server, but demo defaults mention a built-in secret fallback (local demo path).

### Explicitly checked for but not present
- No Express/FastAPI/Flask/Starlette inbound apps.
- No inbound webhook receiver path.
- No DB connection/query layer in runtime web paths.

## 3) Findings Summary

## Result
`no issue found in inspected code` for the target classes (SQLi / XSS / auth-session flaws / common web vulns) within the audited scope.

### A. SQL Injection
- No SQL driver usage (`sqlite3`, SQLAlchemy, psycopg, etc.) in inspected Web/HTTP paths.
- No dynamic query construction sinks identified.

### B. XSS / HTML Injection / Template Injection
- `server.js` returns JSON only; no template rendering.
- `tools/demo_web.py` uses one HTML output (`pig_panel`) composed from fixed internal strings and emoji, not direct user reflection.
- User input (`intent`, `transcript`) is emitted to text/code components, not inserted into raw HTML templates.

### C. Auth / Authorization / Session
- `server.js` explicitly does not execute privileged voice flow; it returns `501 not_implemented` for `/api/voice/run`.
- Security-sensitive session/handshake enforcement remains in Python canonical flow (`po_echo.voice_orchestration`) and is not bypassed by Node gateway.
- No auth bypass endpoint was found in the Node gateway/demo surface.

### D. Other common web vulns
- SSRF: no inbound URL-fetch endpoint from user input.
- Open Redirect: no redirect endpoints.
- Path traversal / arbitrary file read-write via HTTP: no such route.
- Command injection: `spawnSync` in `server.js` executes a fixed command and fixed source string, not user-provided arguments.
- Unsafe deserialization/eval/exec in web paths: not found.
- CORS: not configured because no browser-cross-origin API design is implemented in current gateway surface.

## 4) Evidence-based notes / bounded confidence

- This is a code-level audit of repository content at the time above; it does not prove runtime infrastructure hardening (reverse proxy, firewall, TLS termination, hosting policy).
- Webhook exfiltration risk depends on who can set process environment vars. In this repo, secrets are expected to be injected via env and are not hardcoded.
- Demo files are treated as demo paths; no production deployment manifest is included in this audit scope.
